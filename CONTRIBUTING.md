# Как Вносить Изменения В StackMIREA

Спасибо за вклад в StackMIREA.

Этот репозиторий представляет собой static-first платформу документации для IT-дисциплин. Вкладом в проект могут быть:

- новые учебные материалы
- исправления и актуализация существующей документации
- улучшения UI и UX
- улучшения поиска и навигации
- улучшения content pipeline, валидации и CI
- исправления опечаток, битых ссылок, метаданных и авторства материалов

## Что Прочитать Перед Началом

Перед открытием pull request желательно ознакомиться с файлами:

- [README.md](./README.md)
- [CODE_OF_CONDUCT.md](./CODE_OF_CONDUCT.md)
- [SECURITY.md](./SECURITY.md)

## Структура Проекта

Основные директории:

- `docs/` - исходные материалы, которые редактируются вручную
- `.cache/content-manifest.json` - build-time manifest, который используют приложение, поиск и валидатор
- `resources/` - датасеты и дополнительные артефакты
- `public/` - публичные ассеты, включая `search-index.json`
- `scripts/` - сборка content manifest, валидация и сборка поискового индекса
- `.github/workflows/` - CI и деплой

## Локальный Запуск

Требования:

- `Node.js >= 20`

Установка зависимостей:

```bash
npm ci
```

Запуск в режиме разработки:

```bash
npm run dev
```

Локальный сервер будет доступен на `http://localhost:3000`.

## Рекомендуемый Процесс Работы

1. Сделайте fork репозитория и создайте ветку от `main`.
2. Вносите изменения с понятной и ограниченной областью.
3. Если меняете документацию, редактируйте `docs/`; manifest пересобирается командой `prepare:content`.
4. Запустите нужные проверки локально.
5. Откройте pull request с понятным описанием и результатами проверки.

## Изменения В Контенте

Если вы меняете учебные материалы:

1. Добавьте или обновите файл в `docs/<track>/...`
2. Запустите:

```bash
npm run prepare:content
```

3. Затем проверьте контент:

```bash
npm run validate:content
```

Пайплайн проекта устроен так:

1. `docs/` является редактируемым источником
2. `npm run content:manifest` собирает `.cache/content-manifest.json`
3. `npm run search:build` пересобирает `public/search-index.json` из manifest
4. `npm run prepare:content` выполняет оба шага

## Изменения В Коде

Если вы меняете код, UI, поиск или build pipeline, запустите:

```bash
npm run lint
npm run typecheck
npm run build
```

Если изменение затрагивает рендер документации или обработку контента, дополнительно выполните:

```bash
npm run prepare:content
npm run validate:content
```

## Рекомендации Для Pull Request

Старайтесь, чтобы pull request был:

- небольшим и удобным для ревью
- сфокусированным на одной задаче или одной логической теме
- хорошо описанным
- проверенным локально

В описании PR желательно указать:

- что изменилось
- зачем это было нужно
- как вы это проверили
- скриншоты, если менялся интерфейс
- какие страницы, треки или разделы затронуты, если менялся контент

## Стандарты Для Документации

При редактировании материалов:

- сохраняйте понятную и воспроизводимую структуру
- отдавайте предпочтение практическим и проверяемым примерам
- добавляйте frontmatter-поля `title`, `description`, `order` и `author`, где это уместно
- указывайте язык в code fence
- проверяйте ссылки, якоря и файловые ссылки
- не добавляйте непроверенный или потенциально небезопасный внешний контент

## Коммиты

Строгий формат коммитов не обязателен, но сообщения должны быть:

- понятными
- достаточно конкретными
- удобными для чтения в истории проекта

Примеры:

- `docs(python): add practice 6 notes`
- `search: improve topic matching for AI notebooks`
- `ui: refine docs sidebar mobile behavior`

## Что Могут Попросить На Ревью

Сопровождающие могут попросить доработки, если есть проблемы с:

- корректностью
- качеством контента
- ссылками и якорями
- качеством метаданных
- соответствием структуре проекта
- стабильностью сборки и CI
- безопасностью зависимостей, контента или пайплайна

## Безопасность

Если вы нашли потенциальную уязвимость, не открывайте публичный issue с подробностями.

Используйте процесс из [SECURITY.md](./SECURITY.md).

## Лицензирование

Отправляя вклад в проект, вы соглашаетесь, что:

- изменения в коде распространяются под лицензией MIT
- изменения в контенте распространяются под лицензией `CC BY-NC-SA 4.0`

См.:

- [LICENSE](./LICENSE)
- [CC-BY-NC-SA-4.0](./CC-BY-NC-SA-4.0)

## Если Не Уверены, С Чего Начать

Если непонятно, куда именно вносить изменение:

- откройте issue
- или создайте draft PR с кратким планом

Понятные и аккуратные инкрементальные изменения почти всегда лучше, чем большой неописанный набор правок.
