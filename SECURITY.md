# Политика Безопасности

## Поддерживаемые Версии

Исправления по безопасности и сопровождение предоставляются для актуальной ветки проекта.

| Версия | Поддерживается |
| --- | --- |
| `main` | Да |
| `2.x` | Да |
| `< 2.0` | Нет |

Практически это означает, что ориентироваться нужно на актуальное состояние `main` и текущую опубликованную версию GitHub Pages.

## Как Сообщить Об Уязвимости

Если вы считаете, что нашли проблему безопасности в StackMIREA:

- не публикуйте полный exploit в публичном issue
- используйте приватный security reporting в GitHub, если он включён для репозитория
- если приватный канал недоступен, свяжитесь с сопровождающим через GitHub: <https://github.com/minkinad>

Постарайтесь приложить:

- краткое описание проблемы
- затронутый файл, модуль или зависимость
- оценку возможного ущерба
- шаги воспроизведения или proof of concept
- возможный вариант смягчения проблемы, если он у вас есть

## Что В Этом Проекте Считается Security-Проблемой

Для данного репозитория к security-вопросам относятся:

- уязвимости в зависимостях production-сайта или build pipeline
- небезопасный рендер MDX или HTML
- риски инъекций в scripts или GitHub Actions workflows
- вредоносное или небезопасное выполнение контента во время сборки или рендера
- утечки секретов, токенов или deployment credentials
- supply chain проблемы, влияющие на репозиторий или CI/CD

Для StackMIREA особенно важны доступность и целостность контента, потому что проект строится вокруг content pipeline и статической публикации.

## Что Обычно Не Является Security-Проблемой

Как правило, не считаются security-репортами:

- опечатки
- битые ссылки
- отсутствие метаданных
- ошибки вёрстки
- содержательные ошибки без влияния на безопасность
- feature requests

Такие вопросы лучше оформлять как обычные GitHub issues или pull requests.

## Как Мы Обрабатываем Репорты

Сопровождающий старается:

1. подтвердить получение сообщения
2. проверить, является ли проблема реальной уязвимостью
3. оценить серьёзность и область влияния
4. подготовить исправление или временную mitigation-меру
5. при необходимости раскрыть детали после исправления ответственным образом

Срок реакции может зависеть от доступности сопровождающего, но добросовестные сообщения рассматриваются серьёзно.

## Текущая Модель Безопасности

Проект публикуется как статический сайт в GitHub Pages, что уменьшает поверхность атаки по сравнению с обычным backend-приложением. При этом безопасность всё равно зависит от:

- npm-зависимостей
- компиляции MDX
- scripts для синхронизации и валидации контента
- GitHub Actions workflows

Поэтому регулярное обновление зависимостей, аккуратная работа с контентом и дисциплина в CI остаются важной частью security-модели проекта.
